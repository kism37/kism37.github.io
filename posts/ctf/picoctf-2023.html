Forensics

Who Is It Forensics

The challenge gave us an email file and asked us to identify whose mail server it actually originated from. The key was in the email headers — a raw IP address 173.249.33.206 buried in the Received chain.

I ran a WHOIS lookup on that IP via whois.com and pulled the registrant name directly. That was the flag.

⚑ Flag: PicoCTF{Wilhelm Zwalina}

Lesson: email header analysis is a real-world skill. OSINT on IP addresses — who owns it, what ASN, what org — is bread and butter in threat intelligence work.

Find And Open Forensics

Two files: a password-protected flag.zip and a .pcap. The password was hidden in the packet capture.

Opened the pcap in Wireshark and filtered through the data streams. Found a Base64-encoded string in the packet data, decoded it via CyberChef, and used the result to extract the zip. 

# Decode Base64 in CyberChef → get zip password
# Unzip with recovered password
unzip -P <decoded_password> flag.zip
⚑ Flag: picoCTF{R34DING_L0Kd_fil56_succ3ss_cbf2ebf6}

MSB Forensics

A corrupted-looking image with a hint in the name: MSB — Most Significant Bit. Classic steganography technique where data is encoded in the most significant bits of pixel values rather than the least significant (LSB) which tools like steghide target.

Used stegonline, uploaded the image, selected MSB bit order, and extracted the hidden data as readable ASCII. Downloaded the output file and grepped for the flag. 

cat Ninja-and-Prince-Genji-Ukiyoe-Utagawa-Kunisadaflag.dat | grep picoCTF
⚑ Flag: picoCTF{15_y0ur_que57_qu1x071c_0r_h3r01c_06326238}

Unforgotten Bits Forensics

This one took me a full day. A disk image — the kind of challenge where you have to build an investigation layer by layer rather than find a single trick.

Loaded the disk image into Autopsy for filesystem analysis. Key findings:

  • IRC logs in /home/yone/irclogs/ — contained steghide credentials
  • .bmp images in /home/yone/gallery/ — the steghide targets
  • Encrypted .txt.enc files extracted from images using openssl aes-256-cbc
  • A final image (7.bmp) with a different password — had to brute force it

The 7.bmp password was a League of Legends champion name combination. The IRC logs had half the password (yasuoaatrox) and notes in /home/yone/notes/ hinted at the other half. I generated a full wordlist of LoL champion name combinations with a Python script, then prepended the known prefix and brute-forced with steghide. 

# Generate champion name combinations
with open('champions.txt') as f:
    champions = [c.strip() for c in f.readlines()]

with open('wordlistsLOL.txt', 'w') as f:
    for i in range(len(champions)):
        for j in range(i+1, len(champions)):
            f.write(champions[i] + champions[j] + '\n')
# Decrypt with recovered key/IV/salt from Autopsy
sudo openssl aes-256-cbc -d \
  -in ledger.1.txt.enc -out out1.txt \
  -K a9f86b874bd927057a05408d274ee3a88a83ad972217b81fdc2bb8e8ca8736da \
  -iv 908458e48fc8db1c5a46f18f0feb119f \
  -S 2350e88cbeaf16c9

The final unlock came from a binary string in the disk image that decoded with a golden ratio base cipher — found by following browsing history hints inside the image.

⚑ Flag: picoCTF{f473_53413d_40405b89}
⚠ This challenge is a masterclass in breadcrumb-following. Every artifact pointed to the next. The lesson: don't skip anything in forensics — the IRC log that looked like noise was the key.

Cryptography

Power Analysis: Part 2 Crypto

Power analysis attacks exploit the correlation between an AES encryption key and the power consumption during SubBytes operations. We're given 100 trace files, each containing a plaintext and a power trace.

Step one: parse all trace files and build a traces.json in the format expected by existing solve scripts. 

import json

plaintexts, traces = [], []

for i in range(100):
    filename = f'trace{str(i).zfill(2)}.txt'
    pt, trace = open(filename).read().split('\n')
    pt = bytes.fromhex(pt.split(' ')[-1])
    trace = eval(trace.split("Power trace: ")[-1])
    plaintexts.append(pt)
    traces.append(trace)

out = [{"pt": list(pt), "pm": trace} for pt, trace in zip(plaintexts, traces)]
with open("traces.json", "w") as f:
    f.write(json.dumps(out))

Then ran a Correlation Power Analysis (CPA) script adapted from a Google CTF 2022 writeup. First attempt returned an incorrect flag — found a better implementation on GitHub, stripped a few incompatible lines, and ran it locally (the webshell kept killing the process).

⚑ Flag: picoCTF{2f4981b159a0a78a5e222bc537f894ae}

Power Analysis: Part 1 Crypto

Same technique as Part 2, but the traces come from a live server rather than pre-recorded files. Connected to the server, sent 100 crafted plaintexts, collected the power traces, built traces.json, and ran the same CPA solve script.

⚑ Flag: picoCTF{65cce0eab280e39d12625c7315b03fa1}

Power Analysis: Warmup Crypto

The warmup was actually the hardest of the three to script — 4 hours of work. The server leaks a single bit per query. You send a 16-byte plaintext, it returns a leakage result (7 or 8), and the bit value (0 or 1) encodes information about the XOR of one AES Sbox lookup with a key byte.

To recover each of the 16 key bytes, I had to send 256 plaintexts varying one byte position at a time, collect the leakage values, map 7→0 and 8→1, then brute-force the key byte by finding which value is consistent with the Sbox outputs across all 256 plaintext guesses. 

from pwn import remote

data1 = b""
data3 = b"000000000000000000000000000000"
fo = open("output.txt", 'wb')

for i in range(256):
    r = remote('saturn.picoctf.net', 64600)
    r.recvuntil("Please provide 16 bytes of plaintext encoded as hex: ")
    data2 = str(hex(i))[2:].encode().zfill(2)
    r.sendline(data1 + data2 + data3)
    r.recvuntil("leakage result: ")
    fo.write(r.recvline().rstrip() + b"\n")
fo.close()

Repeated this 16 times — shifting data1 forward and shrinking data3 each round to target each key byte position. Recovered all 16 key bytes as decimals, converted to hex via CyberChef.

⚑ Flag: picoCTF{9dac9d914154db052d7ce1a110fbb3aa}

Reverse Engineering

Virtual Machine 0 Rev

Two files: a .dae (3D model) and an input.txt containing numbers. The 3D model was a gear assembly — opened in Blender to analyse the gear ratio.

Deleted the outer blocks to expose the axles. Counted the teeth: 40 teeth on the red axle, 8 on the blue — gear ratio of 5:1. Multiplied each number in input.txt by 5, converted the results to binary, then decoded binary to ASCII to get the flag.

⚑ Flag: picoCTF{g34r5_0f_m0r3_3537e50a}

Virtual Machine 1 Rev

Same format as VM0 but with a three-gear compound system — significantly more involved. Connected to a live server that gives an input number (changes on restart), loaded the .dae in Blender.

Calculated the compound gear ratio using the formula for adjacent gear trains. After working through it on paper, I got a multiplier of 9359. Multiplied the server's input number by 9359, submitted the result, and got the flag.

⚠ The instance died mid-solve, changing the input number. Always note your server input before doing long calculations offline.
⚑ Flag: picoCTF{m0r3_g34r5_3g4d_1fe58bda}

Closing Thoughts

Two weeks, a dozen challenges, 1st in Africa. The challenges that took the longest — Unforgotten Bits and Power Analysis Warmup — were the ones that taught the most. Forensics rewards patience and methodical enumeration. Power analysis is a real hardware attack technique compressed into a CTF problem. The gear VM challenges were genuinely creative.

Playing with team 8h037 made the hard challenges shorter and the easy ones more fun. CTFS are better with people who can hold the thread when you lose it.
